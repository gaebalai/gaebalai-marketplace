# cc-roundtable — 사용 예시

다분야 전문가 패널의 구조화된 토론으로 의사결정·리뷰·기획을 다각적으로 평가합니다.

## 사이트 / 코드 리뷰

```
이 사이트(https://example.com) 원탁회의 형식으로 리뷰해줘. UX·접근성·성능·SEO·보안 관점에서.
```

또는:

```
~/projects/landing/index.html 코드를 다분야 전문가 5명이 토론해서 평가해줘. 디자인-개발-컨버전 관점.
```

## 기술 선정

```
프론트엔드 프레임워크 선정 토론해줘 (React vs Vue vs Svelte).
우리 팀 상황: 백엔드 5명/프론트 2명, MAU 1만, B2B SaaS, 6개월 안에 출시.
```

오케스트레이터가 기술 매니저 / 시니어 프론트 / 백엔드 / DX 전문가 / DevOps 패널을 자동 구성, Phase 2 토론에서 대립점(생태계 vs 학습 비용 vs 번들 크기) 정리.

## 사업 / 조직 결정

```
신규 채용 1명 — 백엔드 vs 데이터 엔지니어 중 어느 쪽이 6개월 ROI 큰가?
다분야 전문가들과 토론으로 평가. Devil's Advocate 1명 포함 필수.
```

또는:

```
이번 분기 마케팅 예산을 30% 삭감하기로 했어. 어느 채널부터 줄이는 게 맞는지 원탁회의 해줘.
현재 채널: 페북 광고 40%, 검색 광고 30%, 콘텐츠 마케팅 20%, 인플루언서 10%.
```

## 심도 조절

| 표현 | 매핑 |
|---|---|
| "가볍게 평가해줘" / "슬쩍 봐줘" | Quick (Phase 2 건너뜀) |
| (지정 없음) | Standard (최대 2라운드) |
| "철저하게" / "깊이" / "Deep" | Deep (모든 참조 파일 로드) |

## 패널 크기

스킬이 자동 결정하지만 사용자가 명시적으로 지정도 가능:

```
이 결정에 직접 관련된 전문가 7명 + Devil's Advocate 1명 패널로 토론해줘.
```

## Phase 0 의제 재정의 활용

토론에 들어가기 전에 의제 자체를 재정의해야 할 수 있습니다.

```
"우리 회사 OKR 시스템을 어떻게 개선할까?" — 이 의제로 토론 시작 전에 의제 자체부터 검증해줘.
"개선"의 정의가 모호한 것 같은데.
```

스킬이 Phase 0에서 "OKR 시스템의 어떤 측면을 개선하는가? 측정·소통·정렬·동기 중 무엇을 우선?"으로 재구성한 뒤 사용자 승인 받고 진행.

## 결과물 구조

```
roundtable_<ts>/
├── phase0_setup.md      # 의제·전문가 선정·기준
├── phase1_independent/  # 각 전문가의 독립 분석 (병렬)
├── phase2_debate.md     # 대립점 중심 토론 라운드 (Standard/Deep만)
├── phase3_synthesis.md  # 통합 + 소수 의견 명시
└── phase4_report.md     # 사용자용 최종 보고
```
