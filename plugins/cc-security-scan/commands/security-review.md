# security-review

**역할**: 차분 기반 정적 분석 스킬 (PR 단위 · 개발 중에 사용)

이 스킬은 `git diff` 로 변경된 파일만을 대상으로 한다.
전체 파일의 정적 분석은 `/full-scan` 이 담당한다.

> **스코프의 진정성 있는 공개**: 이 스킬은 변경 차분만을 검사하기 때문에, 기존 코드에 잠재된 취약성은 검출하지 않는다. 전체 파일을 대상으로 하고 싶은 경우에는 `/full-scan` 을 사용할 것.

## 전제 조건

- git 리포지터리 내에서 실행할 것
- 서버 기동은 불필요 (정적 분석만)

---

## 절차

### Step 1: 변경 차분 수집

1. 디폴트 브랜치 (`main` / `master` / `develop` 중 하나) 를 `git symbolic-ref refs/remotes/origin/HEAD` 로 검출하고, `git diff {default}...HEAD` 로 변경 파일 일람과 차분을 취득한다
2. 변경 파일 수 · 추가 행 수를 보고한다
3. **스코프를 명시한다**: "이 리뷰는 X 파일의 변경 차분만을 대상으로 합니다" 라고 전한다

### Step 2: 리포지터리 컨텍스트 조사 (Phase 1)

변경 파일을 분석하기 전에, 기존의 안전한 패턴을 파악한다:

1. 기존 인증 · 인가 미들웨어의 구현을 확인한다
2. 기존 입력 밸리데이션 · 새니타이즈 패턴을 확인한다
3. 프로젝트의 인증 모델을 파악한다 (미들웨어 인증 · 리버스 프록시 인증 · 앱 내 인증 등)
4. 변경 파일이 건드리고 있는 도메인 (API 라우트 · 데이터 액세스 층 · 인증 등) 을 특정한다

### Step 3: 차분 비교 분석 (Phase 2)

변경된 코드를 기존의 안전한 패턴과 비교한다:

1. 새로운 코드가 기존의 보안 패턴에서 벗어나지 않았는지 확인한다
2. 새로운 공격면 (새로운 API 엔드포인트 · 외부 입력 접수 개소) 을 특정한다
3. 기존의 인가 체크를 우회하는 코드가 추가되어 있지 않은지 확인한다

### Step 4: 취약성 어세스먼트 (Phase 3)

각 변경 파일에 대해, 이하 카테고리로 취약성을 검사한다.

**검출 대상 (>80% 신뢰도가 있는 것만 보고):**

- **인젝션**: SQL / NoSQL / 명령 / 템플릿 인젝션, 패스 트래버설
- **인증 · 인가**: 인증 우회, 인가 미비, 세션 관리의 결함, JWT 의 잘못된 구현
- **암호 · 시크릿**: 하드코딩된 인증 정보 · API 키, 약한 암호 알고리즘
- **XSS**: `dangerouslySetInnerHTML` / `bypassSecurityTrustHtml` 등의 위험한 메서드만
- **데이터 노출**: PII · 비밀번호 · 토큰의 로그 출력, 센시티브 데이터의 의도하지 않은 반환

**제외 룰 (보고하지 않음):**

- DoS · 리소스 고갈 · 레이트 제한
- React / Angular 의 통상 컴포넌트에서의 XSS (`dangerouslySetInnerHTML` 미사용의 경우)
- 클라이언트 사이드의 인증 체크 결여 (백엔드가 책임을 진다)
- AI 시스템 프롬프트에 대한 사용자 입력 혼입
- 환경 변수 · CLI 플래그를 경유한 공격 (신뢰됨으로 취급)
- 문서 파일 (Markdown)
- 기존 의존성의 기지(旣知) CVE (`/full-scan` 이 담당)
- 로그 스푸핑
- 패스만 제어할 수 있는 SSRF (호스트 · 프로토콜을 제어할 수 있는 경우에만 보고)
- UUID 는 랜덤이며 추측 불가능하다고 가정

### Step 5: 거짓 양성 필터링

신뢰도 8 이상의 발견만 보고한다. 각 발견에 대해 확인한다:

1. 구체적인 공격 경로가 존재하는가 (이론상이 아니라 실제로 악용 가능한가)
2. 실제 보안 리스크인가 (베스트 프랙티스의 결여가 아닌가)
3. 구체적인 코드 개소와 재현 절차가 있는가

### Step 6: 보고서 저장 · 이력 인덱스 업데이트

1. **날짜가 들어간 파일명으로 보고서를 저장한다**
   - 저장 위치: `./security-reports/YYYY-MM-DD-security-review.md`
   - 날짜는 실행 시의 ISO 8601 형식 (예: `2026-05-01`)
   - 디렉터리가 존재하지 않는 경우에는 작성한다

2. **`./security-reports/index.md` 에 1 행 추가 기록한다**
   - 파일이 존재하지 않는 경우에는 헤더부터 작성한다
   - 추가 기록 포맷:
   ```
   | YYYY-MM-DD | security-review | <브랜치명> | <변경 파일 수> 파일 | 0 | <High 건수> | <Medium 건수> | <CI 결과> | [보고서](YYYY-MM-DD-security-review.md) |
   ```
   - CI 결과: High 이상의 발견이 0 건이면 `✅ pass`, 1 건 이상이면 `❌ fail`

## 출력 포맷

```markdown
# 보안 리뷰 결과

**스코프**: <변경 파일 수> 파일의 차분만 (전체 파일 분석은 /full-scan 을 사용)
**대상 브랜치**: <브랜치명>

## Vuln N: <취약성 종별>: `<파일 경로>:<행 번호>`

* Severity: High / Medium
* Confidence: <0.8~1.0>
* Description: <구체적인 문제의 설명>
* Exploit Scenario: <공격 시나리오>
* Recommendation: <수정 방법>
```

**심각도 기준:**
- **High**: 직접 악용 가능한 RCE · 데이터 누출 · 인증 우회
- **Medium**: 특정 조건 하에서 중대한 영향을 가져오는 취약성

## 이 스킬로 커버할 수 없는 영역

| 영역 | 권장 수단 |
|------|---------|
| 변경 차분 이외의 기존 코드의 취약성 | `/full-scan` |
| 의존성의 기지(旣知) CVE | `/full-scan` |
| 배포 후의 런타임 거동 | `/security-scan` |
| 비즈니스 로직의 결함 (사양 지식이 필요) | 침투 테스트 |
| 인프라 · 클라우드 설정 | 인프라 담당자 리뷰 |
