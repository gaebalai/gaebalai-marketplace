# Iteration Log Template

반복 간 추이를 기록하는 템플릿. 매 반복마다 누적해 가며 수렴/발산/종료를 판단한다.

## 헤더 (반복 시작 시 1회 작성)

```markdown
# <대상 프롬프트명> Empirical Tuning Log

- 시작일: YYYY-MM-DD
- 환경: Claude Code / Claude.ai
- 평가자 모델: <모델명>
- 시나리오:
  - A (median): <라벨> — <1줄 요약>
  - B (edge): <라벨> — <1줄 요약>
  - C (hold-out): <라벨> — <1줄 요약, 사용 시점 명시>
- 평가 루브릭: 20점 (skill용 / 기술문서용 / 일반프롬프트용 중 택)
```

## 반복별 기록 (매 반복마다 누적)

```markdown
## 반복 N (YYYY-MM-DD)

### 시나리오별 결과

| 시나리오 | 정확도 | critical | tool_uses | duration_ms | 신규 불명확점 수 |
|---|---|---|---|---|---|
| A | <%> | ✓/✗ | <M> | <D> | <개> |
| B | <%> | ✓/✗ | <M> | <D> | <개> |
| C | <%> | ✓/✗ | <M> | <D> | <개> |

### 점수표 변동 (해당하는 경우)

| 축 | 이전 | 이번 | Δ | 주된 원인 |
|---|---|---|---|---|
| 환경 전제 | 1 | 2 | +1 | OS+버전+사전조건 명시 |
| 코드 완전성 | 1 | 1 | 0 | 변동 없음 |
| ... | ... | ... | ... | ... |
| **합계** | **N** | **N+Δ** | **+Δ** | — |

### 사전 견적 vs 실측

- 사전 견적: +<N>pt (대상 축: <축 이름>)
- 실측: +<N>pt
- 변동 패턴: 보수적 적중 / 상회 / 제로 흔들림 / 견적대로
- 견적 외 변동 원인: <있으면 기술>

### 적용한 수정

<본문에 반영한 수정 내용 1개>

### 임곗값 사전 언어화 (제로 흔들림 방지)

<수정 전에 언어화한 "이 수정이 충족하는 임곗값 문언">

### 다음 반복 수정 대상 (불명확점 풀에서 선정)

<선정한 1개>: <선정 이유>

### 종료 판정 체크

- 신규 불명확점 = 0? <Yes/No>
- 정확도 ≤+3pt 변동? <Yes/No>
- steps ±10% 이내? <Yes/No>
- duration ±15% 이내? <Yes/No>
- (hold-out 투입 시) 직전 평균 대비 -15pt 이상 떨어지지 않음? <Yes/No>

→ 종료 판정: 수렴 / 발산 / 계속 / 종료(충분)
```

## 종료 시 마무리

```markdown
## 최종 추이

| 반복 | 점수 | Δ | 주요 변동 |
|---|---|---|---|
| 1 (베이스라인) | <N>/20 | — | — |
| 2 (<수정 라벨>) | <N>/20 | +<N> | <축 1, 4> |
| 3 (<수정 라벨>) | <N>/20 | +<N> | <축 6> |
| ... | ... | ... | ... |

최종 점수: <N>/20 (<재현 곤란/시도 가능/가능/공개 가능/표준 가능> 존)
종료 사유: 수렴 / 종료(충분) / 발산(구조 재설계 필요)
총 반복 횟수: <N>
주요 학습:
- <이번 튜닝에서 얻은 견지 1>
- <이번 튜닝에서 얻은 견지 2>
- <empirical-prompt-tuning 본체에 피드백할 학습>
```

## 발산 시 처리

3회 이상 반복해도 신규 불명확점이 줄지 않으면:

```markdown
## 발산 판정 (반복 N에서)

연속 <N>회에 걸쳐 신규 불명확점이 줄지 않음:
- 반복 <N-2>: <불명확점 수> (잔존: <항목>)
- 반복 <N-1>: <불명확점 수> (잔존: <항목>)
- 반복 <N>: <불명확점 수> (잔존: <항목>)

→ 패치로는 못 고친다. 본문 구조 재설계 필요.

재설계 방향 후보:
- <후보 1: 본문 분할 / references 분리 / 결정 트리 재구성>
- <후보 2>
- <후보 3>
```

## 자료실 연결

이 로그 자체도 누적되면 메타 자료가 된다. 여러 skill을 튜닝한 후 다음 패턴이 보일 수 있다:

- "환경 전제 0→2 점프는 항상 1회로 끝난다"
- "축 5(설정파일)와 축 8(프로젝트 전제)은 같이 움직이는 경우가 많다"
- "tool_uses 15+ 시그널은 보통 references 분리 미비를 의미한다"

이런 메타 패턴이 축적되면 다음 skill 작성 시 사전 체크리스트가 된다.

## 저장 위치 권장

- Claude Code: `~/.claude/eval-logs/<skill-name>/iteration.md`
- Claude.ai: Artifact로 누적 관리

여러 skill을 동시에 튜닝하는 경우 skill별로 디렉터리 분리.
