# Dispatch Prompt Template

서브에이전트 또는 외부 세션에 던지는 프롬프트의 기본 템플릿. `<...>` 부분을 채워서 사용한다.

---

```
당신은 <대상 프롬프트명>을 백지 상태로 읽는 실행자입니다.
이 글의 작성자가 누구인지, 어떤 의도로 썼는지 추측하지 마십시오.
오직 본문에 명시된 내용만으로 작업하십시오.

## 대상 프롬프트

<프롬프트 본문 또는 파일 경로>

## 시나리오

<상황 설정 1단락>

## 요건 체크리스트

1. [critical] <최저 라인 항목 1>
2. [critical] <최저 라인 항목 2>
3. <일반 항목 1>
4. <일반 항목 2>
5. <일반 항목 3>

## 태스크

1. 대상 프롬프트에 따라 시나리오를 수행하고 산출물을 생성하십시오.
2. 본문에 명시되지 않은 부분은 자기 판단으로 채우되, 채운 내용을
   "재량 보완"으로 보고하십시오.
3. 해석에 망설인 문구는 "불명확점"으로 보고하십시오.
4. 같은 판단을 다시 한 경우 "재시도" 횟수를 보고하십시오.
5. 종료 시 아래 보고 구조로 응답하십시오.
```

## 채우기 가이드

### `<대상 프롬프트명>`

평가 대상의 식별 가능한 이름. 예:
- `~/.claude/skills/conventional-changelog/SKILL.md`
- `/release-notes` (SlashCommand명)
- `2026 prompt engineering trends 발표문 초안`

### `<프롬프트 본문 또는 파일 경로>`

- 짧으면 본문 그대로 inline
- 길면 파일 경로만 (서브에이전트가 읽을 수 있는 환경일 때)
- Claude.ai 외부 세션에 던질 때는 반드시 inline

### `<상황 설정 1단락>`

3~6줄. 너무 짧으면 추측이 들어가고, 너무 길면 본문 베껴쓰기가 된다.
시나리오 설계 가이드(`scenario-design.md`) 참조.

### `[critical]` 태그

요건 5개 중 최소 1개 (권장 2~3개)에 부여. critical 항목이 모두 ○일 때만 성공.

## 변형: 강조 추가

서브에이전트가 본문 내용을 추측으로 보완하는 경향이 강한 경우, 다음 강조를 추가:

```
**중요 — 추측 금지**:
본문에 명시되지 않은 내용을 "이런 의도였을 것이다"라고
추측으로 보완하지 마십시오. 명시되지 않은 모든 결정은
"재량 보완"으로 보고하고, 어떤 근거로 그 값을 골랐는지 함께
적으십시오. "보통 이렇게 한다"는 업계 관행을 채용한 경우에도
재량 보완에 포함하십시오.
```

## 변형: tool_uses 강조

`tool_uses` 측정이 중요한 경우 (구조적 결함 검출용), 다음 추가:

```
**도구 사용 보고**:
태스크 수행 중 사용한 도구(Read, Write, Bash 등)를 모두
나열하고, 각 사용 횟수를 보고하십시오. 본문이 자기 완결적이지
않아 별도 파일을 읽어야 했다면 그 파일명도 함께 보고하십시오.
```

## 변형: hold-out 모드

hold-out 시나리오에서는 다음을 추가하여 과적합 검사 의도를 명시:

```
**hold-out 검사 모드**:
이 시나리오는 본문 작성 시점에 명시적으로 다루어지지 않은
변형 케이스입니다. 본문이 다른 시나리오에 과적합되어 있다면
이 시나리오에서는 정확도가 크게 떨어질 것입니다. 평소대로
정직하게 보고해 주십시오.
```
