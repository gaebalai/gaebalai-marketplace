# 20점 평가 루브릭

원문에서 사용한 10축 × 2점 = 20점 평가 기준의 템플릿. 평가 대상 종류에 따라 축을 선택/변형하여 사용한다.

## 핵심 원칙

**축 이름만 보고 수정하지 말 것.** 각 점수의 임곗값 문언을 한 글자 한 글자 읽고, 수정 전에 "이 수정이 임곗값의 어느 문언을 충족하는가"를 사전 언어화한다. 임곗값 문언을 안 보고 수정하면 +0pt가 나온다.

## Skill 평가용 10축 (재현성 중심)

각 축 0~2점, 총 20점.

### 축 1: 환경 전제 명시
- 0점: 어느 OS / 어느 도구 / 어느 버전을 전제하는지 기재 없음
- 1점: 일부만 명시 (예: OS만, 또는 도구만)
- 2점: OS / 도구 / 버전 / 사전 설치 항목까지 모두 명시

### 축 2: 코드 완전성
- 0점: 코드/프롬프트 단편만, 전체 흐름 추측 필요
- 1점: 핵심 부분 완성형, 일부 호출 예시 누락
- 2점: 복사-실행 가능한 완성형 1세트 inline 또는 직링크

### 축 3: 명령 정확성
- 0점: 실행 명령 기재 없음
- 1점: 일부 명령 기재, 전제 조건/플래그 누락
- 2점: 사용자 환경에서 그대로 실행 가능한 명령 + 전제 조건 명기

### 축 4: 버전 의존성
- 0점: 버전 의존성 기재 없음
- 1점: 메이저 버전만 명시
- 2점: 메이저+마이너 또는 LTS 정책까지 명시, 미충족 시 동작 명기

### 축 5: 설정 파일 예시
- 0점: 설정 파일 예시 없음
- 1점: 일부 키만 발췌
- 2점: 최소 동작 보장 전문(全文) 1개 inline 또는 직링크

### 축 6: 기대 출력 명시
- 0점: 무엇이 출력되는지 기재 없음
- 1점: 출력 예시 일부, 메타데이터 추출 방법 없음
- 2점: 출력 형식 + 메타데이터(usage, 메트릭 등) 추출 방법 명시

### 축 7: 에러 대처
- 0점: 실패 경로 언급 없음
- 1점: 1~2건 언급
- 2점: 주요 실패 케이스 3건 이상 + 대처 명시

### 축 8: 프로젝트 전제
- 0점: 디렉터리 구조 / 파일 배치 암묵
- 1점: 일부 명시 (단일 파일 위치만 등)
- 2점: tree 출력 또는 명시적 배치도 + 어디에 무엇이 있어야 하는지

### 축 9: 링크 건전성
- 0점: 링크 깨짐 / 부재 / 인증 필요한 비공개 자원
- 1점: 일부 링크 살아있음
- 2점: 모든 외부 참조 링크가 공개 가능 + 생존 확인됨

### 축 10: 저자 의존 지식
- 0점: 저자만 가진 사내 지식/도구/문맥에 의존
- 1점: 일부 외부 공개 자원에 매핑됨
- 2점: 모든 의존 지식이 공개 자원으로 추적 가능

## 점수 구간 해석

| 구간 | 의미 |
|---|---|
| 0~6 | 재현 곤란 존. 본인 환경 외에서 동작 보장 불가 |
| 7~11 | 재현 시도 가능 존. 시행착오로 어떻게든 가능 |
| 12~15 | 재현 가능 존. 표준 환경이면 무난히 동작 |
| 16~18 | 공개 가능 존. 어느 정도 검색하면 재현 가능 |
| 19~20 | 표준 가능 존. 거의 모든 환경에서 그대로 동작 |

## 축 변형: 일반 프롬프트 평가용

skill 평가 외 일반 프롬프트 평가에서는 축을 변형:

- 축 1 → "출력 형식 명시" (어떤 형태로 응답해야 하는지)
- 축 5 → "예시 입출력" (few-shot 예시 충분성)
- 축 8 → "역할/페르소나 명시"
- 축 10 → "도메인 지식 의존도"

## 축 변형: 기술 문서/블로그 글 평가용

원문 부록에서 사용한 형태 그대로:

- 축 1: 환경 전제
- 축 2: 코드 완전성
- 축 3: 명령 정확성
- 축 4: 버전 의존
- 축 5: 설정 파일
- 축 6: 기대 출력
- 축 7: 에러 대처
- 축 8: 프로젝트 전제
- 축 9: 링크 건전성
- 축 10: 저자 의존 지식

이 케이스에서 hold-out은 글의 사상 부분이 아닌 핸즈온 재현 부분에 집중.

## 점수 변동 패턴

원문에서 검증된 4가지 변동 패턴:

1. **보수적 적중** (예: +1pt 예측 → +1pt) — 단일 축 단일 임곗값 충족
2. **상회** (예: +2pt 예측 → +4pt) — 1개 구조적 정보가 여러 축 동시 파급
3. **제로 흔들림** (예: +2pt 예측 → 0pt) — 축 이름만 보고 수정, 임곗값 문언 미충족
4. **견적대로** (예: +2pt 예측 → +2pt) — 임곗값 문언 사전 언어화 후 수정

**제로 흔들림을 피하려면**: 수정 전 사용자에게 "이 수정이 어느 축의 어느 점수의 어느 임곗값 문언을 충족하는가"를 명시적으로 답하게 한다.

## 사용자에게 평가 결과 제시할 때

```markdown
| 축 | 점수 | 변동 | 주된 원인 |
|---|---|---|---|
| 환경 전제 | 0 → 1 | +1 | 도구 명시 추가 |
| 코드 완전성 | 1 → 1 | 0 | 변동 없음 |
| ... | ... | ... | ... |
| **합계** | **6 → 9** | **+3** | — |
```

각 반복마다 이 형식으로 제시하여 추이를 가시화한다.
