# 토론 룰

> 구조화된 토론을 운영하기 위한 룰 집.
> 학술적 지견 (MAD 연구, 델파이 기법, Groupthink 회피)에 근거.

## 지나친 동의 문제 (Sycophancy)에 대한 대책

LLM 에이전트는 인간과 마찬가지로 "지나치게 동의하는" 경향이 있습니다.
MAD 연구 (2023-2025)에서 반복적으로 확인되고 있습니다.

### 효과가 있는 대책

| 대책 | 구조 |
|---|---|
| **대립 역할의 명시적 할당** | "당신은 반대 의견을 말하는 역할"이라고 구조에 편입 |
| **독립 의견 생성 페이즈** | 다른 에이전트의 의견을 보기 전에 자신의 의견을 내게 함 |
| **다수파 정보를 전달하지 않음** | "3명 중 2명이 찬성"과 같은 정보는 소수 의견을 억제함 |
| **요약 차분만 전달** | 전체 의사록이 아닌, 대립점의 요약만 전달 |

### 효과가 없는 대책 (사용하지 말 것)

| 대책 | 왜 효과가 없는가 |
|---|---|
| "정확성을 최우선으로 해주세요" | 연구 (2025)에서 효과 없음~역효과로 확인 |
| 자기 신고의 확신도에 의한 가중 | LLM은 토론 시작 시 72.9%, 토론 후 83.3%의 과도한 확신을 가짐 |
| 3라운드 이상의 토론 | Degeneration of Thought: 초기 에러가 고정화됨 |
| 동일 프롬프트의 복제 | Self-Consistency (다수결) 이상의 효과 없음 |

## 소수 의견의 보호 메커니즘

### 왜 보호가 필요한가

MAD 연구의 중요한 발견: **다수파의 압력은 독립적인 수정을 억제한다**.
즉 "3명 중 2명이 찬성"이라는 정보를 전달하면, 나머지 1명이 올바른 의견을 바꿔버립니다.

역사적 사례: 가톨릭 교회가 Devil's Advocate를 폐지한 후, 시성 심사의 질이 저하되었습니다.

### 보호 룰

1. **투표 결과를 보여주지 않고 반론을 요청**: "다른 견해가 있다"고만 전달하고, 누가 다수파인지는 전달하지 않음
2. **소수 의견을 명시적으로 리스트업**: Phase 3에서 소수 의견 체크는 생략 불가
3. **"기각하는 이유"의 기술을 강제**: 소수 의견을 무시하는 경우, 구체적인 이유를 써야 함
4. **의심스러운 경우 채택**: 기각 이유가 불충분하면, 소수 의견을 리포트에 포함

### 소수 의견 체크의 절차

> **주**: 소수 의견 체크의 판정 기준은 SKILL.md Phase 3-3의 2x2 매트릭스
> (근거 유무 × 영향 크기)가 정(正)입니다. 이하의 절차는 그 상세한 실시 방법을 나타냅니다.

```
1. 모든 에이전트의 의견에서, 다수파와 다른 의견을 추출
2. 각 소수 의견에 대해:
   a. 구체적인 근거가 있는가? → 없으면 기각 가능 (이유: 근거 없음)
   b. 다른 에이전트가 이 점에 직접 반론하고 있는가?
      → 반론이 구체적이면 기각 가능 (이유: [반론 요약])
      → 반론이 추상적이면 채택
   c. 이 소수 의견이 옳았다고 할 때의 영향은?
      → 영향이 크다면 채택 (근거가 약하더라도)
3. 채택된 소수 의견은 "소수 의견 섹션"에 기재
4. 기각된 소수 의견은 기각 이유와 함께 기록 (Deep만)
```

## 수렴 조건의 판정

### 수렴의 정의

"이 이상 토론해도 새로운 지견이 나오지 않는" 상태.

### 판정 방법

다음 중 어느 하나를 만족하면 수렴으로 판정:

1. **전원 합의**: 모든 에이전트가 같은 결론에 도달
2. **신규 논점 없음**: Round N의 출력에, Round N-1에 없는 논점이 포함되지 않음
3. **하드 캡**: Round 2에 도달

### Quick 심도의 특별 룰

Quick에서는 Phase 2 (토론)를 건너뜀. 독립 분석의 결과를 직접 통합.
대립이 있어도 토론하지 않고, 오케스트레이터가 가중치로 해결합니다.

---

## 가중치 부여 방법

자기 신고의 확신도는 사용하지 않습니다. 대신 다음 기준으로 가중치를 부여합니다.

### 기준 1: 근거의 구체성

| 레벨 | 예 | 가중치 |
|---|---|---|
| 높음 | 코드 예, 데이터, 사양서 인용 포함 | 가장 중시 |
| 중간 | 구체적인 사례나 경험에 기반 | 표준 |
| 낮음 | "일반적으로", "베스트 프랙티스로서" 등의 추상적 주장 | 낮음 |

### 기준 2: 전문 영역과의 일치도

해당 논점에 가장 관련된 전문가의 의견을 중시합니다.
예: 보안 논점 → 보안 전문가의 의견이 가장 무거움.

### 기준 3: 반론에 대한 응답 품질

반론에 대해 구체적으로 응답한 의견을 중시합니다.
반론을 무시하거나, "그래도 내 의견이 옳다"만으로 응답한 의견은 경시합니다.

## 오케스트레이터의 행동 규범

오케스트레이터의 역할은 Phase에 따라 다릅니다.

### Phase 1-2: 프로세스 관리자

토론의 내용에는 관여하지 않고, 프로세스 제어에 전념합니다.

**해야 할 것:**
- 의제와 평가 기준을 명확히 한다
- 각 에이전트에게 적절한 정보를 전달한다
- 대립점을 정확히 요약한다 (아래의 요약 품질 룰에 따름)
- 소수 의견을 보호한다
- Devil's Advocate의 비판 품질을 관리한다 (아래의 DA 품질 관리 룰 참조)

**해서는 안 되는 것:**
- 자신의 의견을 토론에 넣는다
- 다수파 정보를 에이전트에게 전달한다
- DA에게 합의 경향이나 다수파의 방향성을 전달한다 (DA의 독립된 비판을 오염시키기 때문)
- 에이전트에게 "정답을 내주세요"라고 지시한다
- 3라운드 이상의 토론을 한다

### Phase 3: 통합자

사전 정의된 기준 (근거의 구체성·전문 영역 일치도·반론 응답 품질)에 근거해
판단을 내리고, 최종적인 추천을 냅니다.

**해야 할 것:**
- 가중치 기준을 기계적으로 적용한다
- 소수 의견 체크를 생략하지 않는다
- 판단의 이유를 명시한다 ("왜 이쪽을 채택했는가"를 반드시 기술)

**해서는 안 되는 것:**
- 기준에 없는 독자적 판단축을 가져온다
- 소수 의견 체크를 생략한다

## 대립점의 요약 품질 룰

Phase 2에서 오케스트레이터가 대립점을 요약할 때, 다음을 지킵니다.
요약이 이 스킬 전체의 품질의 싱글 포인트 오브 페일러(single point of failure)임을 자각합니다.

### 요약에 포함해야 할 정보의 최저 기준

- 원래 주장의 **결론**과 **근거**를 모두 포함한다 (결론만의 요약은 불가)
- 구체적인 데이터·사례의 인용은 생략하지 않는다
- 요약에 의해 한쪽 주장이 유리/불리하게 보이는 프레이밍을 하지 않는다

### Deep 심도에서의 추가 룰

- 요약과 함께, 원래 출력의 해당 부분을 인용으로 전달한다 (요약에만 의존하는 것을 완화)

## Devil's Advocate의 품질 관리 룰

Devil's Advocate의 비판 품질은 오케스트레이터가 관리합니다.
페르소나의 자기 규율에만 의존하지 않습니다.

### 과잉 비판의 판정 기준

다음 중 어느 하나에 해당하는 경우, 비판이 과잉 또는 저품질이라고 판단:

- 구체적인 근거나 실패 시나리오를 동반하지 않는 추상적 비판이 3개 이상
- 다른 모든 전문가의 모든 발언에 대해 반대하고 있음 (비판의 초점이 맞지 않음)
- 같은 비판을 다른 표현으로 반복

### 개입 방법

- Phase 2에서 과잉 비판을 검지한 경우: "가장 구조적인 약점 1개로 좁혀서, 구체적인 실패 시나리오와 함께 재분석해주세요"라고 재지시
- Phase 3의 가중치에서: 근거 없는 비판은 명시적으로 낮은 가중치로 한다 (다른 전문가와 같은 기준을 적용)

### 비판이 불충분한 경우의 판정 기준 (하한)

다음 중 어느 하나에 해당하는 경우, 비판이 형해화(형식적이어서 기능하지 않음)되었다고 판단:

- 다른 전문가 발언의 재서술에 머물러, 독자적인 비판적 시점이 없다
- "잠재적인 리스크", "고려해야 할 점이 있다" 등의 지적에만 머물러, 구체적인 실패 시나리오가 없다
- Quick 심도에서 DA의 비판이 모든 전문가의 합의점과 모순되지 않는다 (비판으로 기능하지 않는 조짐)

### 개입 방법 (비판 불충분의 경우)

"만약 이 제안/대상이 최악의 형태로 실패한다면, 그 원인을 하나의 구체적 시나리오로
묘사해주세요"라고 재지시합니다.

## Phase 2에서의 대립점 전달 방식

### 좋은 예

```
당신의 분석에서는 "모바일 퍼스트 레이아웃이 적절"이라고 했지만,
다른 전문가는 "데스크톱 사용자가 70%를 차지하므로 데스크톱 우선이 적절"이라고
분석하고 있습니다.

이 대립에 대해, 당신의 관점에서 다시 분석해주세요.
의견을 바꾸는 경우도, 유지하는 경우도, 구체적인 근거를 제시해주세요.
```

### 나쁜 예

```
다른 3명의 전문가는 데스크톱 우선에 찬성하고 있습니다.
당신의 의견을 재고해주세요.
```

→ 다수파 압력으로 소수 의견이 사라집니다.
