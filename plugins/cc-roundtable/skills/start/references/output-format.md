# 출력 포맷

> 원탁회의 리포트의 출력 템플릿과 기술 룰.

## 최우선 룰

**소스에 기반하지 않은 내용을 생성하지 않는다.**
에이전트의 분석 결과에 기반하지 않은 "정리"나 "보충"을 추가하지 않습니다.
섹션을 채우기 위해 정보를 만들어낼 바에는, 그 섹션을 생략합니다.

---

## 필수 섹션

다음 섹션은 심도에 관계없이 반드시 포함합니다.

### 1. 이그제큐티브 서머리

3-5행으로 결론을 기술. 바쁜 독자가 여기만 읽어도 전체상을 파악할 수 있을 것.

```markdown
## 이그제큐티브 서머리

[가장 중요한 발견을 1행으로]
[전체적인 평가를 1행으로]
[최우선 권장 액션을 1행으로]
```

### 2. 전문가 패널

참가한 전문가의 일람.

```markdown
## 전문가 패널

| 전문가 | 역할 | 종합 평가 |
|---|---|---|
| [이름/역할] | [전문 분야] | ★★★★☆ |
| [이름/역할] | [전문 분야] | ★★★☆☆ |
| Devil's Advocate | [비판의 초점] | — |
```

Devil's Advocate는 "종합 평가"를 내지 않습니다 (비판이 역할이므로 평가는 부적절).

### 3. 합의 사항

모든 전문가가 일치한 점. 대립 없이 전원이 같은 결론에 도달한 사항을 정리합니다.

```markdown
## 합의 사항

- [합의점 1]: [근거의 요약]
- [합의점 2]: [근거의 요약]
```

합의 사항이 없는 경우 "전원이 일치한 점은 없었습니다"라고 명기합니다.

### 4. 주요 쟁점과 결론

대립이 발생한 논점별로, 토론의 경위와 결론을 기재합니다.

```markdown
## 주요 쟁점과 결론

### [논점 1의 타이틀]

- **결론**: [채택된 견해를 1-2문장으로]
- **찬성 측 근거**: [구체적으로. 어느 전문가의 분석인지 명기]
- **반대 측 근거**: [구체적으로. 어느 전문가의 분석인지 명기]
- **판단 이유**: [왜 이쪽을 채택했는가. 가중치 부여의 근거]
```

Quick 심도에서 Phase 2를 건너뛴 경우는 "토론"이 아닌
"다른 견해"로 정리합니다.

### 5. 소수 의견

1명만 지적했지만 중요한 점. **이 섹션은 생략 불가.**

```markdown
## 소수 의견 (놓쳐서는 안 될 지적)

### [지적 내용]
- **지적자**: [전문가 역할]
- **내용**: [구체적인 지적]
- **근거**: [전문가가 제시한 근거]
- **대응**: [채택/기각] — [이유]
```

소수 의견이 없는 경우 "소수 의견은 없었습니다"라고 명기합니다.

### 6. 권장 액션

우선순위가 포함된 행동 제안.

```markdown
## 권장 액션

1. **[최우선]** [액션] — [근거가 된 전문가의 분석]
2. **[높음]** [액션] — [근거]
3. **[중간]** [액션] — [근거]
```

### 7. 토론의 한계

이 토론에서 커버할 수 없었던 관점, 추가 조사가 필요한 점.
사용자가 다음에 무엇을 해야 할지에 대한 시사.

```markdown
## 토론의 한계

- [커버할 수 없었던 관점 1] — [왜 커버할 수 없었는가]
- [추가 조사가 필요한 점] — [구체적으로 무엇을 조사해야 하는가]
```

## 조건부 섹션

다음은 해당하는 경우에만 포함합니다.

### Devil's Advocate의 지적 (Deep만)

Devil's Advocate가 특히 예리한 지적을 한 경우, 독립 섹션으로 기재합니다.

```markdown
## Devil's Advocate의 시점

[비판자가 지적한 구조적인 약점·리스크의 요약]
```

### 토론 프로세스 (Standard / Deep)

토론의 투명성을 담보하기 위한 메타 정보.

```markdown
## 토론 프로세스

- **심도**: [Quick/Standard/Deep] — [판정 이유]
- **실행 방식**: [서브 에이전트 병렬 / Agent Teams 직접 토론 / 순차 실행(fallback)]
- **폴백**: [해당하는 경우] 순차 실행으로 전환한 이유와, 사용자에 대한 통지 결과
- **Phase 2 참가자의 선정 이유**: [각 대립점에 대해, 왜 그 전문가를 선정했는가]
- **수렴 판정**: [어느 조건에서 수렴으로 판정했는가]
- **설계상의 한계**: 서브 에이전트 방식에서의 토론은 델파이 기법에 가까운 구조화된 리뷰이며,
  에이전트 간의 동적인 대화 (MAD 연구가 전제로 하는 스테이트풀한 토론)와는 성질이 다릅니다
```

### 세컨드 찬스의 결과 (Deep만)

Phase 3의 세컨드 찬스의 결과. 추가 지적이 없었던 경우도 생략하지 않고 기재합니다.

```markdown
## 세컨드 찬스 (최종 확인)

[추가 지적이 있는 경우]
- [추가 지적]: [내용과 대응]

[추가 지적이 없는 경우]
추가 지적은 없었습니다 (N명이 사전 커밋과 통합 결과의 일치를 확인).
```

## 기술 룰

### 투명성

- 각 결론에 "어느 전문가의 분석에 근거하는가"를 명기
- 대립의 해결 이유를 명시 ("다수결로 결정했다"는 불가. 근거의 질로 판단)
- 토론의 한계를 정직하게 기술

### 구체성

- 추상적인 평가 ("대체로 양호")가 아닌, 구체적인 지적을 포함
- 권장 액션은 실행 가능한 수준까지 구체화
- "~해야 한다"뿐만 아니라 "왜"를 함께 기술

### 불확실성의 표현

확신도에 따라 문체를 구분해 사용:

| 확신도 | 문체 | 사용 장면 |
|---|---|---|
| 높음 | "~이다", "~이 필요" | 전원 합의 + 구체적 근거 있음 |
| 중간 | "~라고 생각된다", "~이 권장된다" | 다수파 합의 or 조건부 결론 |
| 낮음 | "~의 가능성이 있다", "~을 검토해야 한다" | 소수 의견 or 근거가 한정적 |
